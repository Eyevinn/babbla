# Topics Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let a digest be narrowed to a thematic slice (a "Topic") so the summary covers only matching changes and stays silent when none match this period.

**Architecture:** A `Topic` (name + description) attaches to a digest config (`DigestConfig` for per-project, `SubscriptionDigest` for shared). The digest runner injects a topic-scoping preamble into its single summarization call and normalizes a `NOTHING_RELEVANT` reply to an empty string. The two digest actions pass the topic through, post only when the summary is non-empty, and advance the watermark regardless. Rides the existing digest pipeline — no new module, action, or wiring.

**Tech Stack:** Python 3.11+, `PyYAML`, `pytest` (async tests run without decorators, `asyncio_mode = "auto"`), frozen dataclasses.

## Global Constraints

- **No new dependencies.**
- **Topic is LLM-scoped, not deterministic** — matching happens inside the existing summarization call; no PR-label/path config.
- **Topic = `name` + `description`, both required** when a `topic:` block is present; missing either → `ValueError` at config load (fail-fast, matching the existing `_parse_digest`/`_parse_cadence_tz` style).
- **Silent-on-empty with watermark advance:** when new changes exist but none match the topic, the runner returns `""`; the action posts nothing but DOES advance the watermark. (The existing "no new changes at all" early-return path is unchanged — no post, no advance.)
- **Back-compatible:** a digest with no `topic:` block behaves byte-for-byte as today; the new post-guard is a no-op for non-topic digests (the summary is always non-empty).
- **Sentinel string is exactly `NOTHING_RELEVANT`.**
- **Personal-digest topics are out of scope** — `PersonalDigestAction` is not modified (it calls `summarize_shared` without a topic; the new `topic=None` default keeps it working).

```python
def _parse_topic(label: str, raw: dict | None) -> "Topic | None":
    if not raw:
        return None
    name = str(raw.get("name", "")).strip()
    description = str(raw.get("description", "")).strip()
    if not name or not description:
        raise ValueError(f"{label}: topic requires both name and description")
    return Topic(name=name, description=description)
```
