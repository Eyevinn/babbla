## Purpose

Take Babbla from a laptop process to an **always-on service**, so a sleeping
laptop no longer drops Asks, and add a **scheduled digest** that posts "what
shipped" to a project's channel. This is the infra remnant of the dissolved
"wall" (ADR 0010) plus a minimal realization of ADR 0008.

Two capabilities, one process:

1. **Always-on Q&A** — the existing read-only Q&A connector, packaged to run
   unattended on a server.
2. **Scheduled digest** — a per-project, deploy-aware summary of recent change,
   generated by the same read-only agent path and posted on a cadence.

## Decisions locked (from the Phase 3 brainstorm)

| Question | Decision |
|---|---|
| Where it runs | A **portable container image**: build → validate locally → ship to **Eyevinn OSC** (an always-on instance with injected secrets + a persistent volume). |
| GitHub MCP in a container | Bundle the **`github-mcp-server` binary** and launch it directly (no docker-in-docker). Read-only invariant preserved. |
| Headless Claude auth (open question #1) | A **single shared `ANTHROPIC_API_KEY`** service key, injected as an OSC secret. Locally, no key is needed (Path B subscription login). **No per-user / per-role auth.** |
| Digest anchor | **Per-project configurable**: `branch` (what landed on the default branch) or `deploy` (a GitHub-readable prod-deploy event). Absent = off. |
| First digest target | **MyTV now** (`branch` anchor — it deploys manually off-GitHub, so no deploy signal). **stream-starter later** (`deploy` anchor — Fastly via GitHub Actions). |

## Non-goals (YAGNI)

- **No per-user or per-role identity.** One shared key per project; every user of
  a channel gets identical answers. Access stays Slack-channel membership (ADR 0007).
- **No private-repo onboarding in this phase.** stream-starter is a private `ITV/`
  repo; its ADR-0007 visibility + PII-redaction handling is deferred until it is
  actually onboarded. We build the `deploy` anchor *mechanism* now and validate it
  on fixtures, not live.
- **No per-project Claude keys built now.** A single service key suffices; the
  agent runner already resolves auth from env. Per-project key resolution is a
  trivial later extension, noted but not built.
- **No new runtime dependency.** Anchor detection reuses the Phase-2 stdlib
  (`urllib`) read-only REST reader pattern; no Fastly/Fly API client, no SDK.
- **No public HTTP endpoint.** Socket Mode dials out; there is nothing to expose.
- **No Bedrock/Vertex** wiring now (possible future auth route; out of scope).

## Architecture overview

```
                       ┌──────────────────────────────────────────┐
   Slack (Socket Mode) │  babbla.app  (one always-on process)      │
   ───────────────────▶│   ├─ Slack handlers ─▶ Orchestrator ─▶    │
        outbound WS     │   │                       AgentRunner ──┐ │
                        │   └─ DigestScheduler (async background) ─┤ │
                        │         tick ▶ due? ▶ anchor ▶ summarize─┘ │
                        │                          │                 │
                        │  read-only MCP: github-mcp-server (binary, │
                        │     stdio, GITHUB_READ_ONLY=1)             │
                        │  SQLite on a persistent volume             │
                        └──────────────────────────────────────────┘
                              container image · runs laptop→OSC
```

The digest is **an Ask the system asks itself on a schedule**: it reuses
`AgentRunner` over the same read-only GitHub path and the same citation
discipline as Q&A.

## Part 1 — Runtime & deployment

### Container image

A single image runs everywhere; only injected env/secrets differ between laptop,
VM, and OSC. Contents: Python 3.12 + the Babbla package + the
`github-mcp-server` binary. **No Docker, no Node** inside the image.

- **GitHub MCP via the bundled binary.** `read_only.py` today launches the MCP
  server with `command: "docker", args: ["run", …, "ghcr.io/github/github-mcp-server", "stdio"]`.
  That cannot run inside a container on a managed platform. Replace it with a
  **configurable launcher** that defaults to the binary on `PATH`
  (`command: "github-mcp-server", args: ["stdio"]`) and can still select `docker`
  for hosts that prefer it. The launcher choice is the only thing that changes;
  the read-only guarantee is unchanged because it comes from the **transport and
  flags, not Docker**: `stdio` (never `http`), `GITHUB_READ_ONLY=1`, and a
  read-scoped token.
- **agentmemory wiring becomes optional.** `build_agent_config` currently always
  wires the agentmemory MCP server at `localhost:3111`. A container has no such
  backend (ADR 0009: optional local enrichment). When `AGENTMEMORY_URL` is unset
  or empty, **omit the agentmemory server and its reader tools** entirely. Node /
  `npx` is then unneeded in the image.

### Auth (resolves open question #1)

Auth is **env-pluggable and unchanged in code**: the Agent SDK uses
`ANTHROPIC_API_KEY` if present, else falls back to the Claude Code subscription
login (Path B).

- **Locally:** no key. Path B uses the developer's existing Claude subscription —
  the same auth the pilot's `-m integration` smoke test already uses.
- **On OSC:** inject a **single dedicated `ANTHROPIC_API_KEY`** service key as a
  secret. Shared by all users of the project; no per-user or per-role identity.
- **Not having a key blocks only the OSC cutover**, never the build or local
  validation.

### State & persistence

A small **SQLite** database on a **persistent volume** holds:

- existing Q&A session-resume IDs (`session_store`);
- new per-channel digest state `{watermark, last_digest_at}`.

Losing the volume degrades gracefully: Q&A follow-ups start fresh; the digest
re-bootstraps (see First run).

### Networking

**Socket Mode** dials out over a WebSocket — no inbound port, no public URL, no
ingress config on OSC. OSC must keep **one instance always running** (the
WebSocket and the scheduler both need a long-lived process; no scale-to-zero).

### Deliverables (runtime)

- `Dockerfile` — Python 3.12 base, fetch + install the pinned `github-mcp-server`
  binary, install the package, entrypoint `python -m babbla.app`.
- `.dockerignore`.
- `docker-compose.yml` (or equivalent) for **local validation** — documents the
  keyless Path-B run and the volume mount.
- `docs/DEPLOY.md` — runbook: local run (no key), local container, OSC deploy
  (image, injected secrets, volume, always-on). Includes the env/secret list and
  required `GITHUB_TOKEN` scopes.

## Part 2 — The digest

### Per-project config

Opt-in, in `config/channels.yaml` under each project. **Absent `digest:` = off**
(no surprise posts).

```yaml
  - name: MyTV
    owner: Wkkkkk
    repo: MyTV
    channel_id: C0XXXXXXXXX     # your Slack channel id
    dm: false
    digest:
      cadence: weekly            # off | daily | weekly
      tz: Europe/Stockholm       # for cadence bucketing
      anchor: branch             # branch | deploy
  # stream-starter (later, when onboarded):
  #   digest:
  #     cadence: weekly
  #     tz: Europe/London
  #     anchor: deploy
  #     deploy: { workflow: "cicd_prod.yml" }   # or: environment: production
```

### Anchors

The anchor decides **what counts as "shipped"** and is read deterministically
over the read-only GitHub REST path (no agent, reusing the Phase-2 reader
pattern):

- **`branch`** — change that **landed on the default branch** since the watermark
  (the commits, and the PRs that produced them). Used by MyTV, which deploys
  manually off-GitHub so no deploy event exists. Honest caveat surfaced in the
  digest and docs: *merged, not confirmed-deployed.*
- **`deploy`** — a **GitHub-readable prod-deploy event** since the watermark,
  configured as either:
  - `workflow: <file>` — a **successful run** of a named deploy workflow
    (e.g. stream-starter's `cicd_prod.yml`, a manual `workflow_dispatch`), or
  - `environment: <name>` — a **deployment to a named Environment** (e.g.
    `production`).

  The event carries a head SHA + timestamp; "what shipped" is the commit/PR range
  between consecutive deploy events. This is ADR-0008's "reached an Environment"
  altitude, sourced from GitHub as the read-only stand-in for the not-yet-built
  Deploy/Release MCP (ADR 0004).

A project whose deploy leaves no GitHub trace and that does not opt into `branch`
simply gets **no digest** — graceful, not an error.

### Scheduler & cadence

A **background asyncio task** in the always-on process ticks on a coarse interval
(≈15 min). Each tick, for every digest-enabled channel:

1. **Due?** Compute the cadence bucket of `now` in the channel's `tz`
   (`daily` = calendar date; `weekly` = ISO year-week). Due iff that bucket is
   later than `last_digest_at`'s bucket. This is deterministic, tz-aware, and
   self-healing: downtime just collapses missed buckets into the next post
   (**catch-up-on-wake**, ADR 0008).
2. **Anything new?** Read the anchor for events since the channel's `watermark`.
   Nothing new → stay quiet; do **not** advance `last_digest_at`.
3. **Summarize.** Compute the shipped range (`watermark_sha..head_sha`) and call
   `AgentRunner` with a digest prompt to produce a **cited** summary over that
   range, read-only.
4. **Post & advance.** Post to the project's `channel_id` (top-level channel
   message, not a thread). Advance `watermark` and `last_digest_at`.

The scheduler runs as a background task on the same event loop; a digest agent
run awaits I/O and does not block Slack Q&A.

### Watermark, first run, skip rules

- **State:** per channel `{watermark, last_digest_at}` in SQLite on the volume.
- **First run** (`last_digest_at` null): initialize the watermark to the latest
  anchor event and post **one bootstrap digest** of a bounded window
  (`branch`: change in the last cadence period; `deploy`: the most recent deploy's
  delta), so the channel sees one digest immediately without dumping all history.
- **Skip:** projects without a `channel_id` (the DM-only pilot project) are
  skipped — digests are a channel broadcast.

### Ops entry point

`python -m babbla.digest --once` runs a single due-check-and-post pass for all
channels (or a named one) and exits. Used for local validation and manual ticks;
it shares all logic with the scheduler.

## Data flow (digest)

```
DigestScheduler.tick(now)
  for each project with digest enabled and a channel_id:
    due = cadence_bucket(now, tz) > cadence_bucket(last_digest_at, tz)
    if not due: continue
    events = anchors.read(project, since=watermark)        # read-only REST, GET-only
    if not events: continue                                # quiet; no advance
    range = (watermark_sha, events.head_sha)
    summary = runner.summarize(project, range)             # AgentRunner, cited, read-only
    poster.post(channel_id, summary)                       # top-level channel message
    store.advance(channel, watermark=events.head_sha, last_digest_at=now)
```

## Testing strategy

Four layers; only the last needs a key.

1. **Unit — no credentials, no network.**
   - `scheduler` due-logic: cadence bucketing across `daily`/`weekly` and tz
     boundaries; first-run; downtime collapses to one post; quiet when no events.
   - `anchors`: `branch` and `deploy` readers over **fixture REST JSON**
     (including stream-starter-shaped `workflow`/`environment` fixtures, since the
     live repo is private and not onboarded); empty/404 → no events.
   - `config`: parse + validate the `digest` block; `absent = off`; bad cadence/
     anchor rejected.
   - `session_store`: watermark get/advance round-trips.
   - `poster`: builds the expected Slack call (client injected).
   - `runner`: builds the digest prompt for a range; returns the agent's text
     (agent injected).
   - **Generalized read-only guard** (`test_read_only_guard.py`): assert the
     invariant — `stdio` present, `http` absent, `GITHUB_READ_ONLY=1`, no write
     tools — for **both** launcher modes (binary and docker); agentmemory tools
     present only when configured.
2. **Local end-to-end — Path B, no key.** `python -m babbla.app` and
   `python -m babbla.digest --once` against MyTV with the developer's Claude
   subscription + read-only `GITHUB_TOKEN` + the local GitHub MCP. Posts a live,
   cited digest. Marked `-m integration`.
3. **Local container (optional).** `docker run` the image to validate packaging;
   auth via mounted host credential or a key. Not required for logic coverage.
4. **OSC cutover.** Needs the injected key; deferred until one exists.

## Security / read-only invariants preserved

- **Read-only is unchanged.** The GitHub MCP stays `stdio` + `GITHUB_READ_ONLY=1`
  + read-scoped token; only the launcher (binary vs docker) changes. The guard
  test is **strengthened** to pin the invariant rather than the launcher.
- **Digest path is read-only too.** Anchor reading is GET-only `urllib`; the agent
  summary uses the same read-only config as Q&A. The only write is the Slack post.
- **Token scopes:** `branch` needs repo `contents`/`metadata` read; `deploy` adds
  `actions:read` (workflow runs) and/or `deployments:read`. Documented in
  `DEPLOY.md`.
- **agentmemory** is off by default in the server image (no contents uploaded
  anywhere — ADR 0009).
